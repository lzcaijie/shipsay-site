# V5_1_TEMPLATE_STANDARD（Shipsay 模板标准完善版草案）

> 目的：这份文档用于把当前 Shipsay 实际运行链路里“准确无误”的部分沉淀为模板标准；允许暂时不全，但不能写错。
>
> 当前阶段：以 `themes/shipsay/` 为母模板基线，先把 Shipsay 扫准、写准，再向 v5.2 收口。
>
> 关键原则：Shipsay 当前可作为例外做小范围校准；从下一个模板开始，必须完全按本标准执行。

---

## 1. 单一目标

后续模板整理的唯一目标是：

**基于“完全准确的 Shipsay”形成一套可复制、可验证、不会误导后续模板的标准。**

优先级固定为：

1. **定义不能错**
2. **不破坏现有正确逻辑**
3. **模板职责清晰**
4. **变量来源准确**
5. **页面显示正常，手机端优先**
6. **在不破坏页面协调性的前提下做统一**

---

## 2. 总原则（必须遵守）

### 2.1 宁可暂时缺失，也不能写错

- 缺失可以后补
- 错误不能进入标准
- 不确定的函数、标签、变量，先标记待确认，不准写死

### 2.2 模板去匹配核心，不是核心去匹配模板

默认顺序：
1. 先核真实基线
2. 先核变量来源
3. 先核页面职责
4. 再决定模板怎么适配

除非用户明确授权，否则不为了迁就模板去动 `app / class / include / configs`。

### 2.3 前台禁止展示内部实现说明

页面前台禁止出现这类文案：
- 默认展示……
- 当前逻辑……
- 本页规则……
- 内部说明……

这些内容只能写在：
- `docs/*.md`
- 必要代码注释
- 开发约定

### 2.4 模板不能把现存例外反写成标准

当前 Shipsay 可以保留少量历史例外，但：
- 例外必须在文档里明确标注
- 例外不能写成“推荐做法”
- 下一个模板不能继续复制这些例外

---

## 3. 文档真源与执行顺序

### 3.1 根目录 `docs/` 是唯一真源

当前标准文档真源固定在：
- `docs/CHANGELOG.md`
- `docs/OPS.md`
- `docs/PROJECT_STRUCTURE.md`
- `docs/V5_1_TEMPLATE_STANDARD.md`
- `docs/VARIABLE_MAP.md`

### 3.2 `shipsay/docs/` 只保留迁移占位

`shipsay/docs/*` 不再放正文，只保留“已迁移”占位，避免旧路径误读。

### 3.3 修改模板前必须先读文档

每次修改模板前，至少先核对：
1. `docs/V5_1_TEMPLATE_STANDARD.md`
2. `docs/VARIABLE_MAP.md`
3. `docs/PROJECT_STRUCTURE.md`

没有先读文档，不进入模板修改。

---

## 4. 文件修改红线

### 4.1 当前 Shipsay 的例外权限

当前阶段因为正在用 Shipsay 反推标准，所以：
- Shipsay 可在**确有必要且已核对清楚**时，小范围修改核心
- 但仍以“不动核心”为优先
- 每次改核心都必须能说明：为什么模板层无法准确收口

### 4.2 从下一个模板开始的固定红线

从下一个模板开始，以及后续所有模板：
- **默认只能修改当前模板目录下文件**
- 默认不能再动：
  - `shipsay/app/*`
  - `shipsay/class/*`
  - `shipsay/include/*`
  - `shipsay/configs/*`
  - `www/configs/*`
- 除非用户明确授权，否则不突破这条红线

### 4.3 没有真实基线，不输出部署代码

如果用户没有给真实最新源码包 / 真实文件：
- 不默认给 A/B
- 不默认指导覆盖部署
- 只能做分析和待确认判断

---

## 5. 结构边界标准

### 5.1 模板层只负责展示

`themes/*` 只负责：
- 页面结构
- 模块展示
- 已准备好的变量输出
- 轻量级局部兜底（如安全链接、默认文案）

### 5.2 app 层负责数据准备

`shipsay/app/*` 负责：
- 页面查询
- 数据筛选
- 页面变量准备
- 模板入口选择

### 5.3 class/include/configs 负责底层与公共逻辑

包括：
- 路由
- URL 规则
- SEO 渲染
- 公共函数
- 缓存 / 数据库 / 错误页等底层逻辑

### 5.4 明确禁止：模板直接查库 / 读缓存 / 拼 SQL

这条必须写进标准。

当前全目录扫描已确认：
- `themes/shipsay/tpl_top.php` 里仍存在直接拼 `$sql`、调 `$redis->ss_redis_getrows()`、调 `$db->ss_getrows()` 的历史写法

这属于：
- **Shipsay 当前现存例外**
- **不是后续模板标准**

后续模板一律禁止复制这类写法。

---

## 6. 页面职责标准

### 6.1 首页 `tpl_home.php`

必须保留：
- 顶部导航与搜索
- 推荐书区
- 核心列表（最近更新 / 最新入库等）
- 站点基础入口

当前扫描补充：
- 首页 SEO 需要单独列为重点核查项
- 首页不能因为“模块补全”随意堆叠排行块

### 6.2 分类页 `tpl_category.php`

必须保留：
- 分类导航
- 书籍列表
- 分页 / 跳转逻辑
- 手机端可读性

### 6.3 详情页 `tpl_info.php`

必须保留：
- 封面 / 书名 / 作者 / 分类 / 状态 / 字数
- 最新章节信息
- 开始阅读 / 查看目录
- 作品简介
- 章节区
- 长尾相关推荐

当前扫描新增要求：
- 章节区标题与真实语义必须一致
- 页面上不能展示内部规则说明文案

### 6.4 目录页 `tpl_indexlist.php`

必须保留：
- 书籍基础信息
- 目录标题
- 开始阅读 / 返回详情
- 章节列表
- 分页逻辑
- 与详情页一致的长尾相关推荐模块

### 6.5 阅读页 `tpl_reader.php`

必须保留：
- 面包屑
- 章节标题
- 作者 / 时间 / 字数等信息
- 正文 `<?=$rico_content?>`
- 上一章 / 下一章 / 返回目录 / 返回详情
- 阅读分页信息（启用时）

### 6.6 搜索页 / 作者页 / 错误页 / 最近阅读页

这些页面也要遵守同一条：
- 变量来源准确
- 前台不暴露内部规则
- SEO 不乱写一套脱离系统链路的逻辑

### 6.7 排行聚合页 / 榜单详情页要分开定义

本轮扫描已确认：
- `tpl_top.php` 更偏聚合页
- `tpl_rank.php` 更偏榜单详情页
- `shipsay/app/top.php` 会根据模板存在情况，在 `tpl_rank_list.php / tpl_rank.php / tpl_top.php` 之间选择详情模板

所以标准里必须明确：
- 聚合页职责
- 详情页职责
- 兼容 fallback 顺序

不能再把“排行页”笼统写成一页。

---

## 7. SEO 标准（先定义准确性，再扩展完整性）

### 7.1 SEO 优先服从系统现有链路

当前已确认系统 SEO 链路：
- `shipsay/seo.php`
- `shipsay/configs/seo_tpl.php`
- 模板页内 `ss_seo_render($page)`

模板默认不能脱离这套链路，自行写死另一套 title/keywords/description 规则。

### 7.2 页面级最低要求

当前扫描确认：
- `tpl_info.php`：SEO 相对完整（canonical、mobile-agent、JSON-LD、OG）
- `tpl_indexlist.php`：已有 canonical、mobile-agent、JSON-LD
- `tpl_reader.php`：已有 mobile-agent、OG
- `tpl_home.php`：明显比详情/目录页薄，后续需重点核查
- `tpl_top.php`：当前走局部 SEO，不属于理想标准写法

因此文档里必须区分：
- 已达标页面
- 需补齐页面
- 历史例外页面

### 7.3 首页 SEO 是后续重点核查项

当前只能写到这一步：
- 首页 SEO 相比详情/目录页更薄
- 这是**已确认现象**
- 但要补哪些项，必须在继续核对实际输出链路后再定

不在本轮把未核实的 SEO 规则写死。

---

## 8. 当前全目录扫描已确认的问题定义

以下几条已经确认，应正式写入标准：

1. **错误：把 URL 占位符当模板运行时变量使用**
2. **错误：把模板局部派生变量当成公共核心变量使用**
3. **错误：模板文件直接查库 / 读缓存 / 拼 SQL**
4. **错误：前台展示内部实现规则说明**
5. **错误：根目录 docs 与 `shipsay/docs` 双真源混淆**
6. **错误：排行榜聚合页与榜单详情页职责未分清**
7. **错误：在未核实真实运行效果前，仅凭局部代码字面推断变量语义**

第 7 条尤其重要：
- 例如章节区变量语义，必须结合真实运行页验证
- 不能只看 `array_slice/array_reverse` 就下结论

## 9. 当前 Shipsay 的“现存例外清单”（允许记录，不允许扩散）

### 9.1 `themes/shipsay/tpl_top.php`
- 当前存在模板层查询数据逻辑
- 这是历史遗留例外，不是标准

### 9.2 `themes/shipsay/tpl_info.php`
- 当前仍有前台内部规则说明文案
- 章节区第二块标题与标准目标存在不一致风险
- 这是待修项，不是标准定义

### 9.3 搜索 helper 定义不统一
- 当前 Shipsay 核心里未确认存在统一 `ss_search_url()` 定义
- 现有模板多采用 `function_exists(...) ? ... : fallback` 兜底
- 这属于“兼容写法现状”，不是已定型的核心 API 标准

---

## 10. 执行顺序（当前阶段固定）

当前阶段的正确顺序固定为：

1. **全目录扫描**
2. **发现问题先回写文档**
3. **优先校准变量 / 结构 / 函数 / 标签的准确性**
4. **把 Shipsay 修成准确基线**
5. **再往 v5.2 收口**

后续如果遇到冲突，统一按这条顺序处理。
