# V5.1_TEMPLATE_STANDARD（Shipsay 模板标准草案）

> 目的：这份文档不是给“人看着热闹”的，而是给后续 **AI / 人工快速判断能不能改、该怎么改、哪些不能乱改** 用的。
>
> 适用范围：`shipsay-site` 分站侧模板标准，当前以 `themes/shipsay/` 作为母模板基线；其它主题（如 `88ds`）可参考，但默认不反向覆盖母模板规则。
>
> 当前状态：**v5.1 草案**。本版以“删减 v5.0 里过硬、容易误改的规则 + 补齐真实链路 + 明确验收方式”为主。

---

## 1. 单一目标

后续模板整理的主要目标只有一个：

**让 AI 或人工在看到源码后，能快速、安全地复制一套新模板，而不是为了改模板把原来正确的逻辑改坏。**

所以本标准优先级如下：

1. **不破坏现有正确逻辑**
2. **变量来源准确**
3. **模板职责清晰**
4. **页面显示正常，移动端优先**
5. **在不破坏页面协调性的前提下做统一**

---

## 2. 核心原则（必须遵守）

### 2.1 不能为了“按标准改”而把原来正确的东西改坏

这条优先级最高。

出现以下情况时，默认 **不改**：
- 页面已经显示正常
- 当前变量链路已经正确
- 当前 SEO 已由系统统一输出
- 当前页面结构与程序职责一致
- 改动只会“更像标准”，但没有真实收益

### 2.2 模板不能写死后台可变链接

凡是后台可改或程序可生成的链接，模板里都不能写死。例如：
- `fake_top`
- `fake_recentread`
- `fake_info_url`
- `fake_indexlist`
- 分类链接
- 搜索链接

### 2.3 SEO 优先服从程序实际输出链路

系统中已经存在：
- `shipsay/seo.php`
- `shipsay/configs/seo_tpl.php`

因此模板默认不能再为了“看起来完整”去硬写：

```html
<title>分站1</title>
<meta name="keywords" content="分站1">
<meta name="description" content="分站1">
```

正确规则：
1. 先复用系统现有 SEO 链路
2. 模板只负责输出已经准备好的 `seo_title / seo_keywords / seo_description`
3. 只有在程序未提供时，才允许模板给极轻的兜底值
4. 不允许模板层强行覆盖公共 SEO 逻辑
5. `tpl_recentread.php` 与 `tpl_error.php` 当前阶段只要求可用，不作为 SEO 优先强化页
6. `tpl_top.php` 可先在模板层做 SEO 兜底，但 `shipsay/seo.php` 与 `shipsay/configs/seo_tpl.php` 的总控联调留到模板整体测试阶段

### 2.4 页面添加内容必须与页面气质一致

加模块不是越多越好。所有新增内容都必须满足：
- 与当前模板风格一致
- 不打断主阅读流
- 不造成空白异常或布局失衡
- 手机端能正常显示

## 5. 公共规则

## 5.1 头部与尾部

### Header 负责
- 站点 logo / 标题
- 搜索入口
- 首页 / 书库 / 完本 / 足迹 等公共入口
- 顶部导航

### Footer 负责
- 底部文案
- 版权区
- 公共统计脚本或站点尾部信息

### 规则
- 模板页统一使用 `__THEME_DIR__` include
- 静态资源统一走：`/static/<?=$theme_dir?>/`
- 搜索入口必须走程序/配置生成的可变链接，不能写死 `/search/`
- 头部导航链接必须优先走 safe 变量（如 `$allbooks_url_safe`、`$full_allbooks_url_safe`、`$recentread_url_safe`、`$rank_entry_safe`），不能直接信任未兜底变量
- 页面全局数据调用数量优先使用双数；若为栅格/多列模块，默认先按双数收口，避免 PC 端出现空白列。
- 模板中未使用、且会干扰链路判断的历史变量应及时清理，避免后续把“死变量”误当成真实标准。

## 5.2 手机端优先

验收时先看手机端，再看 PC 端。

必须满足：
- 不横向撑破
- 不错位
- 不出现大片无意义空白
- 不因加模块影响主体阅读/点击流
- 按钮可点，搜索可用，分页可用

实现方式要求：
- 优先在现有规则上顺着收口
- 优先复用现有选择器与现有结构
- 能不用 `!important` 就不用
- 不能靠“强制覆盖一整块”来掩盖结构问题

### 手机端封面邻接区补充标准（新增）

对首页推荐卡、分类列表卡、搜索结果卡、书架卡、首页分类首卡、详情页/目录页顶部书封区，后续默认遵守：

- 优先把封面区视为独立“封面列”，不要让图片尺寸在同页里忽大忽小
- 同类带图卡片在手机端尽量统一封面宽高比例，默认优先收口到接近 1 : 1.34 的节奏
- 封面图默认使用 `object-fit: cover`，优先避免因为原图比例杂乱导致上下留白或左右挤压
- 右侧文字列必须确保 `min-width: 0`，标题、简介、作者、字数、更新时间按稳定节奏排布
- 优先通过现有选择器微调封面尺寸、gap、line-height、clamp 行数来收口，不为了解决图片周围不协调而新增大块 HTML 结构
- 若首页、分类页、详情页、目录页都出现“封面附近空、字与图贴不齐、图片区太弱”的同类问题，默认按同一套移动端封面列标准处理，而不是每页各写一套

### 手机端通用问题归类（新增）

优先排查并沉淀为通用规则的，不是所有模块，而是**带图片的列表卡片**。

重点包括：
- 左图右文卡片在手机端的错位、挤压、留白、对齐问题
- 图片容器宽高不统一，导致同一列表上下跳动
- 文字容器未设置 `min-width: 0`，导致标题/简介撑破布局
- 标题、简介、作者、字数、更新时间等信息行高不一致
- 按钮、标签、状态条在移动端过宽或过高，造成视觉断裂
- 优先做间距、封面尺寸、标题行高、作者/字数/更新时间一行节奏的微调，不为了一次收口而重写整块结构
- 首页推荐卡、分类列表卡、首页分类首卡、详情/目录顶部书封信息区要尽量沿同一套移动端节奏收口，便于后续模板复用

### 手机端 footer 标准（新增）

底部区域属于全站通用模块，后续默认按“轻量收口”处理，不采用强压覆盖式修法。

标准要求：
- 底部文案整体居中
- 每一行文案在手机端保持居中对齐
- 优先沿用现有 footer 结构调整，不随意重写 HTML
- 优先通过更具体但克制的选择器修正布局，不默认追加大量 `!important`
- footer 的修法要能复用到首页、分类页、详情页、目录页、阅读记录页等全站页面
- 若页面已基本正常，只做轻量微调，不为了“看起来更整齐”继续叠加强覆盖样式

### 分类分页块手机端标准（新增）

分类页手机端分页默认使用“上一页 / 页码选择 / 下一页”的纵向收口方式，要求：

- 三个元素纵向排列
- 按钮宽度统一
- 选择框宽度统一
- 不出现中间大块空白
- 不依赖额外模板结构改造，优先兼容程序当前输出的 `$jump_html_wap`

### 手机端低风险区（补充）

- 纯文字列表
- 章节目录纯文本列表
- 阅读页正文区

这类页面通常问题更少，默认优先少动；手机端优化时，先收带图卡片，再回头校准纯文字模块。

## 5.3 标签一致性

页面中的：
- 标题
- 面包屑
- 按钮文案
- 模块标题
- SEO 标题

必须保持一致，不能页面写“月排行榜”，SEO 却还是“总排行榜”。

## 6. 各页面职责与标准

## 6.1 首页 `tpl_home.php`

### 必须保留
- 推荐书区
- 最近更新/最新入库之类的核心列表
- 顶部导航与搜索
- 站点基础入口

### 不再强制
- **不强制首页必须嵌入完整排行榜模块**

### v5.1 新规则
- 首页只要求提供 **排行榜入口** 即可
- 是否展示完整排行榜块，由模板风格决定
- 如果首页已经平衡，不要为了“模块齐全”硬塞排行块

## 6.2 栏目页 `tpl_category.php`

### 当前目标
- PC 端可分栏
- 手机端单列
- 长标题不撑坏布局

### 规则
- PC 端推荐双列或多列
- 长标题可限制字数 / 两行截断
- 简介区高度要有统一节奏
- 可加入长尾推荐链接，但不能抢主列表视觉

## 6.3 详情页 `tpl_info.php`

### 必须保留
- 封面 / 书名 / 作者 / 分类 / 状态 / 字数
- 最新章节信息
- 开始阅读 / 查看目录
- 作品简介
- 最新章节区
- 长尾相关推荐

### 明确禁止
- 为了“更简洁”把章节区删掉
- 为了“更统一”把相关推荐删掉

### 说明
详情页不是只有“书信息卡片 + 两个按钮”。

## 6.4 目录页 `tpl_indexlist.php`

### 必须保留
- 书籍基础信息
- 目录标题
- 开始阅读 / 返回详情
- 章节列表
- 分页逻辑（如启用分页）

### 规则
- 目录列表不能出现右半边大面积无意义留白
- 章节列表区宽度要贴合主内容
- 手机端目录按钮优先竖排或自然换行
- 目录页长尾相关推荐必须与详情页保持同一套：同标题、同判断条件、同链接字段、同展示字段
- 目录页 SEO 至少要有：canonical、TDK、OG 基础项、BreadcrumbList 结构化数据

## 6.5 阅读页 `tpl_reader.php`

### 必须保留
- 面包屑
- 章节标题
- 作者 / 字数 / 时间 等基础信息
- 正文输出：`<?=$rico_content?>`
- 上一章 / 下一章 / 返回目录 / 返回详情
- 阅读分页信息（如果启用阅读分页）

### 规则
- 正文区必须优先保证阅读体验
- 面包屑、分页、设置按钮不能压主阅读内容
- 不能因为“加功能”破坏正文可读性
- 阅读页 SEO 以当前章节页为准，至少输出：canonical、TDK、OG 基础项、BreadcrumbList 结构化数据
- `og:title / og:description` 优先跟随当前页面 SEO 结果，不再额外拼出一套与 TDK 不一致的文案

## 6.6 搜索页 `tpl_search.php`

### 必须保留
- 搜索表单
- 结果列表
- 结果数量
- 搜索关键词标题一致性

### 规则
- 搜索表单样式要与头部保持一致
- 搜索结果卡片不能比首页/栏目页风格完全脱节

## 6.7 阅读记录 `tpl_recentread.php`

### v5.1 统一规则
- 单独一页，不与排行榜或推荐列表混用
- 头部入口固定保留
- 支持无封面占位
- PC / 手机端使用同一数据逻辑，不是两套含义不同的布局

## 6.8 排行榜 `tpl_rank.php / tpl_top.php / tpl_rank_list.php`

### 统一目标
- 聚合榜入口
- 单榜页
- 榜单切换
- 搜索区
- 手机端自适应

### 新规则
- 后台 `fake_top` 是唯一主入口前缀
- 单榜页链接也必须基于 `fake_top`
- 榜单切换必须自适应
- 不允许模板里继续写死 `/top/` 或 `/rank/`

## 7. v5.0 删减项（v5.1 明确收回）

以下条目在 v5.1 中明确弱化或删除：

1. **首页必须有完整排行榜模块** → 改为“首页必须有排行榜入口”
2. **模板默认补 SEO title/keywords/description** → 改为“优先服从系统 SEO 链路”
3. **为统一而删减详情页章节区/相关推荐** → 禁止
4. **只强调模板齐全，不强调页面职责** → 改为“职责优先”

## 8. 移动端验收标准

每一轮模板修改都必须过这一组检查：

### 首页 / 列表
- 图片、标题、简介、作者信息对齐正常
- 搜索框不压行、不偏移
- 导航不遮挡主内容

### 详情页
- 简介区无异常留白
- 按钮区位置自然
- 最新章节区可正常阅读

### 目录页
- 章节列表宽度正常
- 按钮与标题区不冲突
- 不留大面积空白

### 阅读页
- 面包屑不过长或挤压主内容
- 分页信息位置合理
- 设置按钮不抢正文阅读区

### 排行榜/搜索/足迹
- 手机端可点击
- 搜索框不变形
- 榜单切换可自适应
- 列表排版不溢出

## 9. 修改流程（AI / 人工共用）

每次改模板，按这个顺序：

1. 先确认当前源码包是最新基线
2. 先核变量来源
3. 先核页面职责
4. 先判断当前是否已经正确
5. 再决定改不改
6. 修改后先做基础语法检查
7. 再打包交付

不允许直接“看页面像有问题”就凭印象大改。

## 7. 本轮新增红线（2026-03-07）

### 7.1 准确性优先于完整性

当前以“完全准确的 Shipsay”作为母模板基线。

规则：
- 文档中的函数、标签、变量定义 **可以暂时缺失**
- 但 **不能写错**
- 缺失项后续可以继续补
- 错误定义一旦进入标准，会被后续所有模板继承放大

### 7.2 前台禁止展示内部实现说明

页面前台禁止直接展示以下类型文案：
- 默认展示……
- 当前逻辑……
- 本页规则……
- 模板内部标准说明……

这类内容只能放在：
- `docs/*.md`
- 必要代码注释
- 内部协作说明

### 7.3 模板不能直接承担数据层职责

标准规则：
- `themes/*` 只负责展示
- `shipsay/app/*` 负责准备页面数据
- `class/include/configs` 负责底层逻辑与公共能力

明确禁止：
- 模板中直接拼 SQL
- 模板中直接读 Redis
- 模板中直接查数据库

当前基线已将排行聚合页的数据准备迁回 `shipsay/app/top.php`，`themes/shipsay/tpl_top.php` 只保留展示职责。后续模板必须沿用这种边界，不得再把榜单查询写回模板。


补充要求：
- 搜索页模板不得直接裸输出 `$searchkey`，必须先做安全转义；需要高亮时，也必须在转义后输出。
- 分类页、头部、排行入口这类公共链接必须优先走 `*_safe` 兜底变量，不能直接假定链路变量一定存在。

### 7.4 文档真源固定

唯一真源固定在根目录：
- `docs/CHANGELOG.md`
- `docs/OPS.md`
- `docs/PROJECT_STRUCTURE.md`
- `docs/VARIABLE_MAP.md`
- `docs/V5_1_TEMPLATE_STANDARD.md`

`shipsay/docs/` 只保留迁移占位，不再放正文。

### 7.5 Shipsay 当前例外与后续统一规则

当前阶段：
- Shipsay 正在作为标准反推母模板
- 如遇到已经核对清楚、且不小改核心就无法准确收口的问题
- **允许 Shipsay 小范围例外修核心**

但从 **下一个模板开始**：
- 默认只能修改当前模板目录下文件
- 不再动 `app / class / include / configs` 等核心文件
- 修改模板前必须先阅读 `docs/` 标准文件

### 7.6 章节、排序、混淆链路必须以真实运行结果为准

涉及章节顺序、目录分页、阅读 URL、混淆 ID 时：
- 不能只看局部代码就下结论
- 必须结合真实运行页验证
- 必须以最终页面输出结果校准定义

这条尤其适用于：
- `tpl_info.php`
- `tpl_indexlist.php`
- `tpl_reader.php`
- 对应 `shipsay/app/*.php` 的章节准备链路

## 8. 本轮 SEO 补充标准（2026-03-07）

### 8.1 首页 `tpl_home.php`

首页至少应统一输出：
- `title / keywords / description`
- `canonical`
- `applicable-device / mobile-agent`
- 基础 OG 信息
- `WebSite` 级别结构化数据

### 8.2 分类/作者/排行页

这类可被搜索引擎稳定抓取的列表页，至少应统一输出：
- `title / keywords / description`
- `canonical`
- `applicable-device / mobile-agent`
- 基础 OG 信息
- `BreadcrumbList` 结构化数据

### 8.3 详情/目录/阅读页

详情页、目录页、阅读页已经是核心 SEO 页面：
- 优先沿用现有系统 SEO 链路
- 模板仅做安全兜底和结构统一
- 不允许为“补 SEO”去重写核心逻辑

### 8.4 错误页、阅读记录、搜索页

这几类页面后续可继续细化，但当前优先级低于：首页、分类、作者、排行、详情、目录、阅读。

## 9. 手机端带图区排版补充标准（2026-03-07）

### 9.1 首页 / 分类页带图卡片

手机端带图卡片的默认收口目标：
- 封面与文字顶部对齐
- 标题优先稳定在 1–2 行
- 简介控制在 2–3 行内，不和底部信息互相挤压
- 作者信息与字数 / 更新时间保持同一底部节奏

实现原则：
- 优先沿用现有 `.side_commend / .searchresult / #tempBookcase` 结构
- 优先微调封面宽高、gap、line-height、clamp
- 不为修手机端卡片去重排模板 DOM
- 非必要不使用 `!important` 强压覆盖

### 9.2 首页分类首卡 `sortvisit`

`sortvisit` 首条“左图右文”在手机端容易出现：
- 封面与右侧文案上下节奏不齐
- 作者信息依赖 `<br>` 撑开，导致视觉不稳定
- 简介有效行数偏少，看起来像被截断

标准做法：
- 保持左图右文结构不变
- 标题、作者、简介分成更清晰的三段节奏
- 作者信息单独成行
- 只通过 CSS 收口，不先改模板结构

### 9.3 详情页 / 目录页顶部图文区

手机端 `novel-basic-info` 标准：
- 书封为固定列
- 右侧信息区为自适应列
- 标题、基础元信息、最新章节、按钮区分层清楚
- 作者 / 分类 / 状态 / 字数优先按两列节奏收口

注意：
- 这里的优化属于“图文信息区排版标准”，不是变量标准修改
- 不得为了手机端细调去改动章节、目录、SEO 等核心链路


### 9.4 详情页 / 目录页书封信息区的手机端补充

当作者名、章节名较长时，`novel-basic-info` 手机端不要死守僵硬两列元信息。

默认收口原则：
- 书封区仍保持左图右文，不改模板结构。
- 书名保持单独一层。
- 作者信息优先保证完整可读，必要时单独占一行。
- 分类 / 状态 / 字数可顺势换到下一行，不强求平均分栏。
- 最新章节与时间保留同一区块，但优先可读，不为了“凑整齐”压坏折行。

这条规则的目标不是追求表面完全对称，而是保证手机端图文邻接区的阅读稳定性。

### 9.5 首页分类首卡 `sortvisit` 的手机端补充

首页分类首卡本质上也是“左图右文”带图卡片，只是文案结构更老。

默认收口原则：
- 继续沿用现有 DOM，不为手机端去重排模板结构。
- 封面顶部对齐，右侧文案与封面顶部贴齐。
- 标题、作者、简介必须有明确层级，不能继续依赖 `<br>` 撑视觉。
- 优先通过 `line-height / margin / 封面尺寸 / 文案区宽度` 微调完成适配。
- 不允许用大块强制覆盖把整个首卡“压平”。


### Mobile image-text fit (2026-03-07 addendum)
- 带图卡片在移动端优先固定封面列宽，再让标题、简介、作者与字数/时间构成稳定文字列。
- 首页分类首卡 `sortvisit` 首图条目应与普通带图卡片保持接近的封面宽高比与文字节奏，避免图片过小而文字块过散。
- 详情页、目录页顶部信息区优先采用“固定封面列 + 自适应元信息列”布局，最新章节链接允许单独换行，不强行与时间挤在同一行。
